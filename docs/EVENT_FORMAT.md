# Event & Message Format

This is the **single source of truth** for every message that crosses a service
boundary. The same shapes are produced by the worker, carried over Redis,
persisted by the API, and pushed to the browser over WebSocket. Do not redefine
these per service â€” import/copy from here.

All timestamps are **ISO-8601 UTC strings** (e.g. `2026-06-23T13:59:00.123Z`).
All ids are **UUID v4** strings.

## 1. Camera command (API â†’ Redis â†’ Worker)

Published by the API when a user clicks Start/Stop or updates rules. The worker
subscribes and acts on it. Channel: `camera:commands`.

### Start command

```json
{
  "type": "start",                       // "start" | "stop" | "rules_update"
  "camera_id": "f6c1...",                // UUID of the camera
  "rtsp_url": "rtsp://mediamtx:8554/cam", // present on "start", omitted on "stop"/"rules_update"
  "rules": [                             // present on "start" and "rules_update"
    {
      "id": "r1",
      "classes": ["person", "car"],
      "zone": [{ "x": 0.1, "y": 0.1 }, { "x": 0.9, "y": 0.1 }, { "x": 0.5, "y": 0.9 }],
      "schedule": ["22:00", "06:00"],    // [start, end] or [null, null] for no schedule
      "min_confidence": 0.4,
      "severity": "low",                 // "low" | "medium" | "high"
      "enabled": true
    }
  ],
  "requested_by": "9ab2...",             // user UUID (audit)
  "ts": "2026-06-23T13:59:00.123Z"
}
```

Each rule in `rules` (on both `start` and `rules_update`) may also carry:
- `type`: `"presence"` (default when omitted) | `"tripwire"` | `"dwell"`.
- `direction`: `"in"` | `"out"` | `"both"` â€” tripwire only, which way a crossing counts (`"both"` when omitted).
- `dwell_seconds`: dwell only â€” seconds a tracked object must stay continuously inside `zone` before it fires.

For a `tripwire` rule, `zone` is a **2-point line** (`[{"x","y"}, {"x","y"}]`) instead of the polygon used by `presence`/`dwell` rules.

### Rules update command

```json
{
  "type": "rules_update",                // sent when rules change while camera is running
  "camera_id": "f6c1...",
  "rules": [ /* same shape as above */ ],
  "requested_by": "9ab2...",
  "ts": "2026-06-23T13:59:00.123Z"
}
```

### Stop command

```json
{
  "type": "stop",
  "camera_id": "f6c1...",
  "requested_by": "9ab2...",
  "ts": "2026-06-23T13:59:00.123Z"
}
```

### Discover command (ONVIF scan)

```json
{
  "type": "discover",                      // ONVIF network discovery
  "scan_id": "scan-123...",                // UUID v4, correlates results back to the user
  "user_id": "9ab2...",                    // user UUID (results are private to this user)
  "username": "admin",                     // credential for discovered cameras
  "password": "12345",                     // credential for discovered cameras
  "ts": "2026-06-23T13:59:00.123Z"
}
```

The worker receives this on `camera:commands` and broadcasts a multicast ONVIF probe. Results are published to Redis on `discovery:results` (see Â§7). The command has **no `camera_id`** â€” it's a one-shot scan, not tied to a running camera.

## 2. Detection event (Worker â†’ Redis â†’ API â†’ DB)

Emitted by the worker whenever â‰Ą1 detection matches a rule (subject to
dedup/rate-limit, see Â§5). Delivered via Redis Streams: written to `stream:detections`
by the worker, consumed by the backend using consumer group `vms-backend`. Payload
stored under the `data` field in the stream entry. Persisted to the `alerts`
table verbatim (minus transport fields).

```json
{
  "id": "1f0e...",                       // UUID, generated by worker (idempotency key)
  "type": "detection",                   // detection event (was "person_detected")
  "camera_id": "f6c1...",
  "ts": "2026-06-23T13:59:01.500Z",      // detection time
  "label": "person",                     // class that triggered this detection (e.g., "person", "car")
  "rule_id": "r1",                       // which rule matched this detection
  "severity": "high",                    // "low" | "medium" | "high" from the matched rule
  "confidence": 0.91,                    // max confidence for the matched class in frame [0..1]
  "count": 2,                            // number of objects of the matched class in frame
  "bboxes": [                            // normalized [0..1], origin top-left; all objects of the matched class
    { "x": 0.10, "y": 0.20, "w": 0.30, "h": 0.55, "conf": 0.91, "label": "person" },
    { "x": 0.62, "y": 0.31, "w": 0.18, "h": 0.40, "conf": 0.74, "label": "person" }
  ],
  "frame_w": 1280,
  "frame_h": 720,
  "worker_id": "worker-1"
}
```

## 3. Camera stats (Worker â†’ Redis â†’ API â†’ WS)

Emitted ~once per second per running camera. Not persisted (or persisted as
rolling latest only). Channel: `stats`.

```json
{
  "type": "camera_stats",
  "camera_id": "f6c1...",
  "ts": "2026-06-23T13:59:01.000Z",
  "fps": 24.5,                           // decoded frames/sec, smoothed
  "detections_per_min": 12,              // rolling 60s person-detection rate
  "state": "live",                       // connecting | live | stopped | reconnecting | offline
  "reconnect_count": 3,                  // int, cumulative reconnects for this session
  "last_frame_at": "2026-06-23T13:59:01.000Z"  // iso string|null, time of last decoded frame
}
```

## 4. Camera state change (Worker â†’ Redis â†’ API â†’ WS)

Emitted on every lifecycle transition so the UI tile reflects truth.
Channel: `stats` (same channel, distinguished by `type`).

```json
{
  "type": "camera_state",
  "camera_id": "f6c1...",
  "ts": "2026-06-23T13:59:00.800Z",
  "state": "reconnecting",               // connecting | live | stopped | reconnecting | offline
  "detail": "stream stalled, reconnecting"   // human-readable, optional
}
```

## 5. WebSocket envelope (API â†’ Browser)

Everything the browser receives is wrapped in one envelope so a single socket
multiplexes alerts + stats + state. The client switches on `channel`.

```json
{
  "channel": "alert",                    // "alert" | "stats" | "state" | "clip"
  "data": { /* Â§2, Â§3, or Â§4 payload */ }
}
```

The browser only receives events for cameras owned by the authenticated user;
the API filters server-side before pushing.

## 6. Clip ready (Worker â†’ Redis â†’ API â†’ DB â†’ WS)

Emitted by the worker when an event-clip finishes writing (~POST_ROLL after the
last triggering detection). Delivered via Redis Streams: written to `stream:clips`
by the worker, consumed by the backend using consumer group `vms-backend`. Payload
stored under the `data` field in the stream entry. Persisted to the `clips` table;
the API links it to the alert via `alert_id` and fans it out as a `clip`
envelope. `path`/`thumb_path` are relative to `RECORDINGS_DIR`.

```json
{
  "type": "clip_ready",
  "id": "9c2a...",                       // UUID, worker-generated (idempotency key)
  "alert_id": "1f0e...",                 // the detection/alert that triggered it
  "camera_id": "f6c1...",
  "start_ts": "2026-07-06T13:59:01.500Z",
  "end_ts":   "2026-07-06T13:59:21.500Z",
  "duration_ms": 20000,
  "size_bytes": 1830000,
  "path": "f6c1.../9c2a....mp4",
  "thumb_path": "f6c1.../9c2a....jpg",
  "backend": "local",
  "worker_id": "worker-1"
}
```

The WebSocket envelope (Â§5) `channel` may also be `clip`, whose `data` is the
payload above.

## 7. Discovery results (Worker â†’ Redis â†’ API â†’ WS)

Emitted by the worker in response to a `discover` command (Â§1.4). Channel:
`discovery:results`. The API relays this directly to the user over WebSocket
(opaque envelope).

```json
{
  "scan_id": "scan-123...",                // UUID, echoes the discover command's scan_id
  "user_id": "9ab2...",                    // UUID, echoes the discover command's user_id
  "cameras": [                             // discovered devices, may be empty
    {
      "name": "Front Door",                // device friendly name
      "ip": "192.168.1.100",               // device IP address
      "hardware": "Dahua IPC-HDBW2431E",   // manufacturer + model
      "rtsp_url": "rtsp://192.168.1.100:554/stream1",  // auto-discovered RTSP endpoint
      "error": null                        // null on success, string error message if auth failed
    }
  ],
  "error": null                            // null on success; non-empty string if multicast scan failed entirely
}
```

## Dedup & rate-limit (worker-side, see README)

- **Dedup:** a detection is suppressed if the previous emitted detection for the
  same `camera_id` was < `DEDUP_WINDOW_MS` (default 3000ms) ago **and** the
  person `count` did not increase. Stops 25 alerts/sec from one standing person.
- **Rate-limit:** hard cap of `MAX_EVENTS_PER_MIN` (default 30) persisted alerts
  per camera per minute. Excess detections still update stats but are not stored.

These two knobs keep the `alerts` table and the WS stream sane under continuous
presence.
